# Wove 将来の技術候補メモ

## 位置づけ

このドキュメントは、**現時点では必須ではないが、今後の開発で採用候補になり得るライブラリ・外部サービス** を整理するためのメモである。

目的は以下。

- 将来の選択肢を忘れないようにする
- 今の段階で「採用しない理由」も残しておく
- 必要になったときにすぐ検討を再開できるようにする

**重要:**  
ここに書かれているものは、現時点で採用決定したものではない。  
あくまで「候補一覧」であり、MVPスコープを勝手に広げるためのものではない。

---

# 1. データ取得 / 通信状態管理

## TanStack Query

### 何をしてくれるか

- API通信結果のキャッシュ
- loading / error / success 状態管理
- 再取得（refetch）
- 更新後のキャッシュ更新
- mutation（POST / PATCH / DELETE）管理

### Woveで使いそうな場面

- curricula 一覧取得
- curriculum 詳細取得
- topic 詳細取得
- summary 保存
- topic 編集
- 保存後の再取得や一覧反映

### 利点

- `loading` / `error` を毎回 `useState` で持たなくてよくなる
- API通信まわりの実装が整理される
- 一覧 → 詳細 → 更新 のような画面遷移と相性が良い
- フロント側のデータ取得責務が明確になる

### デメリット

- 学習コストが少しある
- 小さい画面や単発フォームにはやや過剰
- 最初に QueryClient のセットアップが必要

### 現時点の判断

**今すぐ導入しない。**

理由：

- ログインフォームだけなら `useState` で十分
- ただし、一覧取得・詳細取得・更新処理が増えたら、かなり有力な候補

### 導入タイミング候補

- `curricula` 一覧画面のAPI接続時
- `topic` 編集や `summary` 保存が複数画面に増えてきた時

---

## SWR

### 何をしてくれるか

- データ取得とキャッシュ管理
- 軽量な loading / error 管理

### 利点

- TanStack Query より軽量
- Next.js と相性が良い
- 学習コストが比較的低い

### デメリット

- mutation や更新系は TanStack Query より弱め
- Wove のような「取得 + 更新」が多いアプリでは、後々 TanStack Query の方が伸びやすい可能性が高い

### 現時点の判断

**候補としては残すが、第一候補ではない。**

---

# 2. 認証まわり

## Supabase Auth（継続利用）

### 何をしてくれるか

- メール/パスワード認証
- セッション管理
- JWT発行
- social login
- magic link
- OTP

### 利点

- すでに Wove で採用済み
- DB / RLS / Auth が一体で管理できる
- 自前認証の責任を大きく減らせる

### デメリット

- 認証後の画面制御や UX は自分で設計する必要がある
- 専用認証サービスほど UI / DX が強くない

### 現時点の判断

**継続採用。**
Wove ではまず Supabase Auth を使い切る方針でよい。

---

## Google Login（Supabase Auth 経由）

### 何をしてくれるか

- Google アカウントによるログイン

### 利点

- 初回登録の摩擦が下がる
- パスワード管理の負担が減る
- 個人開発で運用コストを減らしやすい

### デメリット

- Google Cloud 側の設定が必要
- コールバックURL管理が必要

### 現時点の判断

**かなり有力。**
公開前〜MVP公開タイミングで導入候補。

---

## Magic Link（Supabase Auth）

### 何をしてくれるか

- パスワードなしログイン
- メールのリンクからログイン

### 利点

- パスワード運用をさらに減らせる
- ユーザー体験が軽い

### デメリット

- メール受信導線に依存する
- 人によっては使いづらい

### 現時点の判断

**候補。**
Google Login と比較して選ぶ。

---

## Clerk

### 何をしてくれるか

- 高機能な認証UI
- social login
- セッション管理
- 認証画面の完成度が高い

### 利点

- 認証UIをかなり省力化できる
- DXが良い
- 個人開発で便利

### デメリット

- Supabase Auth と役割が競合する
- 認証を分離しすぎると構成が複雑になる
- Wove では今の段階では過剰

### 現時点の判断

**今は不採用。**
将来、認証UXが大きな課題になったら再検討。

---

## Auth0

### 何をしてくれるか

- エンタープライズ向け認証基盤
- social login / SSO / 高度な認証管理

### 利点

- 強力で実績がある
- 将来的な認証要件に広く対応できる

### デメリット

- Wove の現段階では重い
- 運用や構成が複雑になりやすい

### 現時点の判断

**今は不要。**

---

# 3. フォームまわり

## React Hook Form

### 何をしてくれるか

- フォーム状態管理
- バリデーション
- 入力値管理
- エラーハンドリング

### Woveで使いそうな場面

- ログインフォーム
- 要約保存フォーム
- topic 編集フォーム
- 将来の設定画面

### 利点

- `useState` を大量に書かなくてよくなる
- フォーム実装が整理される
- Zod と組み合わせやすい

### デメリット

- 最初は少し学習が必要
- 小さいフォームだけなら過剰なこともある

### 現時点の判断

**かなり有力。**
フォームが2〜3個を超えてきたら導入候補。

---

## Zod

### 何をしてくれるか

- 型安全なバリデーション
- フロントとバックで使える入力検証

### 利点

- TypeScript と相性が良い
- フォームと API の入力定義を揃えやすい
- バリデーションルールを明文化できる

### デメリット

- 最初に設計意識が少し必要

### 現時点の判断

**かなり有力。**
Wove とは相性が良い。

---

# 4. UI / 状態 / 補助

## Sonner（Toast）

### 何をしてくれるか

- 保存成功 / 失敗などの通知UI

### 利点

- UXがかなり良くなる
- 要約保存や編集完了のフィードバックに向く

### デメリット

- MVP初期ではなくてもよい

### 現時点の判断

**軽量で導入しやすい候補。**
要約保存体験を磨くタイミングで導入候補。

---

## Zustand

### 何をしてくれるか

- 軽量グローバル状態管理

### 利点

- React Context より扱いやすいことが多い
- UI状態の共有に向く

### デメリット

- 今の Wove ではまだ不要
- サーバー状態は TanStack Query の方が役割に合う

### 現時点の判断

**今は不要。**

---

# 5. 監視 / 分析

## Sentry

### 何をしてくれるか

- フロント / バックのエラー監視

### 利点

- 本番エラーを追いやすい
- 個人開発でも後から効く

### デメリット

- MVP初期では後回し可能

### 現時点の判断

**MVP後〜公開後の候補。**

---

## PostHog

### 何をしてくれるか

- 行動分析
- イベント計測
- 継続率 / 保存率 / 再開率の可視化

### Woveで重要な指標

- 要約保存率
- 再開率
- 離脱ポイント

### 利点

- Wove の価値検証と非常に相性が良い

### デメリット

- 最初から入れるとやや重い
- イベント設計が必要

### 現時点の判断

**MVP後のかなり有力な候補。**

---

# 6. 今の時点での優先順位

## 近いうちに有力

- TanStack Query
- React Hook Form
- Zod
- Google Login（Supabase Auth 経由）

## 少し後で有力

- Sonner
- PostHog
- Sentry

## 今は不要

- Clerk
- Auth0
- Zustand
- SWR（完全に不要ではないが優先度低め）

---

# 7. 現時点の結論

Wove の現段階では、以下を意識しておくのがよい。

### 今すぐ入れなくてよいが、覚えておくべき有力候補

- TanStack Query
- React Hook Form
- Zod
- Google Login

### MVP後に検討しやすい候補

- Sonner
- PostHog
- Sentry

### 現時点で重要な方針

**「便利そうだから入れる」のではなく、必要になったタイミングで最小限導入する。**

Wove では、MVPスコープを守ることが最優先である。
