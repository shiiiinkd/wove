# Wove Phase3 実装方針（web）

Wove の Phase3 では、これまで `wove/api` に実装してきた Hono API を土台に、`wove/web` に最小UIを実装する。

## このフェーズの目的

目的はデザインを作り込むことではなく、以下の一気通貫フローをブラウザ上で成立させること。

- ログイン
- カリキュラム一覧表示
- カリキュラム詳細表示
- トピック詳細表示
- 要約保存
- 保存結果確認
- トピック編集

## 責務分離

- web: UI / 画面遷移 / 表示 / フォーム
- api: 業務ロジック / DBアクセス / バリデーション

Next.js 側に業務ロジックを寄せすぎないこと。

## 実装対象

- ログイン画面
- カリキュラム一覧画面
- カリキュラム詳細画面
- トピック詳細画面
- 要約保存画面
- トピック編集機能

## 各画面の役割

### カリキュラム一覧

- 保存済みカリキュラム一覧を表示
- クリックでカリキュラム詳細へ

### カリキュラム詳細

- タイトル
- 概要
- トピック一覧
- 各トピックの進捗ステータス表示

### トピック詳細

- トピック名
- トピック説明
- 進捗ステータス
- 最新要約
- 要約保存導線
- 編集導線

### 要約保存画面

- 要約入力フォーム
- 保存後はトピック詳細へ戻る

### トピック編集

- 編集対象は title, description
- 保存後はトピック詳細へ戻る

## 遷移原則

変更後は必ず「変更結果が確認できる画面」に戻す。

- 要約保存後 → トピック詳細
- 編集後 → トピック詳細

## データ表示前提

DB 構造は以下を前提とする。

- curricula
- topics
- summaries

特に UI では以下を使う。

- curricula.title / slug / description
- topics.title / description / order_index / status
- summaries.content / is_latest

表示ルール:

- topics は order_index 順に表示
- topic 詳細では is_latest = true の summary を表示
- status は not_started / in_progress / completed をユーザー向け表示に変換する

## 実装順序

1. App Router で必要ページの骨組み作成
2. カリキュラム一覧の実装
3. カリキュラム詳細の実装
4. トピック詳細の実装
5. 要約保存フォームの実装
6. トピック編集機能の実装

## UI方針

- 最初は最小UIでよい
- 取得中心画面はシンプルに実装
- フォーム系のみ client component を使う
- 空状態・エラー状態も最低限対応する
- デザインの作り込みは後回し

## 今回やらないこと

- アプリ内カリキュラム新規作成UI
- トピック手動追加
- 並び替え
- 履歴一覧UI
- 高度なテストUI
- デザインの作り込み

## 完了条件

以下がブラウザ上で通れば Phase3 完了とみなす。

- ログイン後に一覧へ遷移できる
- カリキュラム一覧が見える
- カリキュラム詳細でトピック一覧と進捗が見える
- トピック詳細で最新要約が見える
- 要約を保存できる
- 保存後にトピック詳細へ戻って結果確認できる
- トピック編集できる
