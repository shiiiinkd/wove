# Wove Agent Guide

このリポジトリで作業を行う際は、まずここを読んでください。

## このプロジェクトについて

Wove は、ChatGPTで行った学習を保存・構造化し、継続できるようにする学習管理アプリです。

コア価値：

- 学習の構造化
- 学習の保存
- 学習の再開可能化

このプロダクトは、AIによる生成が主目的ではありません。  
AIは生成を担当し、Woveは保存と継続を担います。

---

## アーキテクチャ

このリポジトリはモノレポ構成を採用しています。

- `apps/web` → Next.js フロントエンド（UI層）
- `apps/api` → Hono バックエンド（業務/API層）
- Supabase → 認証 + Postgres + RLS

### 責務分離

- `web` の役割：
  - UI
  - ルーティング
  - フォーム
  - 表示

- `api` の役割：
  - 業務ロジック
  - バリデーション
  - DBアクセス
  - 認証前提のAPI処理

**業務ロジックを `web` に入れないこと。**

## 開発の基本原則

- MVPのスコープを守る
- 小さく段階的に実装する
- 現在のフェーズを超えた機能追加を勝手に行わない
- 更新処理の後は、必ず結果を確認できる画面に戻す
- 過剰設計よりも、分かりやすさと保守性を優先する

---

## 現在のプロダクトフロー（MVP）

ユーザーの基本的な流れ：

1. ログインする
2. 保存済みカリキュラムを見る
3. カリキュラムを開く
4. トピックを開く
5. 要約を保存する
6. 後日再開する

MVPで扱う機能：

- カリキュラム閲覧
- トピック管理
- 要約保存
- 進捗の可視化

MVPで扱わないもの：

- アプリ内カリキュラム作成UI
- トピック並び替え
- 高度な分析
- 複雑なテストUI

---

## 重要なデータモデル

主要エンティティ：

- `profiles`
- `curricula`
- `topics`
- `summaries`

前提：

- `curricula` はユーザーに属する
- `topics` はカリキュラムに属する
- `summaries` はトピックに属する
- topics は `status` を持つ
- summaries は履歴を持つが、UIでは最新のみ表示する

## 作業の進め方

実装前に：

1. タスクの範囲を理解する
2. 影響範囲のファイルを特定する
3. 小さな実装計画を立てる
4. 不要な機能拡張を避ける

実装時：

- 変更は小さく保つ
- ファイルの責務を崩さない
- まずはシンプルな実装を優先する

実装後：

- 変更ファイルをまとめる
- 影響範囲を説明する
- 未解決のリスクがあれば記載する
